## 学习笔记

### 🔵 AI 线：Prompt 工程核心

| 概念 | 一句话理解 | 工程启示 |
|------|-----------|---------|
| **Instruction** | 给模型的任务规则：角色、目标、边界、输出格式 | 拆成四段：任务目标 / 可用输入 / 禁止行为 / 输出格式与失败格式 |
| **Few-shot** | 放少量示例让模型模仿判断方式和输出格式 | 适合"风格和边界难说清"的任务，但示例要跟 eval 一起维护，协议升级后旧示例会误导 |
| **Structured Output** | 模型按固定结构返回结果（JSON / schema） | 让后续代码能检查、拒绝、记录和回归测试，不是为了好看 |
| **Prompt Injection** | 攻击者通过用户输入/网页/文档让模型忽略原始规则 | 不能只靠"不要被注入"，要：外部内容标记不可信、工具调用前参数校验、敏感动作走 allowlist |

**第一性原理：**
> Prompt 是软约束，不是安全边界。
> 真正的边界必须由代码、权限、校验和审计来承担。

**指令分层原则：**
```
系统规则            ← 最高优先级，不可被用户/文档覆盖
开发者规则          ← 任务定义
用户目标            ← 动态输入
检索内容 / 外部文档  ← 标记为不可信数据，最低优先级
```

**Instruction 实用四段式：**
```
1. 任务目标       — 你是什么角色、要完成什么
2. 可用输入       — 哪些信息可以使用
3. 禁止行为       — 不能做什么、不能假设什么
4. 输出格式       — 正常格式 + 失败/不确定时的格式
```

**Structured Output 示例（交易风险摘要）：**
```json
{
  "action": "explain_transaction / prepare_swap / reject",
  "risk_level": "low / medium / high",
  "requires_human_approval": true,
  "uncertainties": ["不能验证的事实列表"]
}
```

**Prompt Injection 应对方案（从弱到强）：**
1. ❌ 写一句"不要被注入" — 不够
2. ✅ 外部内容标记为不可信数据
3. ✅ 工具调用前做参数校验
4. ✅ 敏感动作走 allowlist + human approval
5. ✅ 密钥、主权限、不可撤销动作不能交给模型

**AI × Web3 中的位置：**
```
Prompt 定义任务和输出格式
  → Context 提供可信数据和来源边界
    → Model 生成解释或候选动作
      → Code 校验 schema 和业务规则
        → Guard / simulation 检查链上影响
          → Human check 确认高风险动作
```

### 🟢 Web3 线：密码学基础

| 概念 | 一句话理解 | 关键认知 |
|------|-----------|---------|
| **Hash** | 把任意数据变成固定长度指纹，验证同一性和完整性 | 不是加密，不能"解密回原文"。改一个字符输出完全不同 |
| **Public Key** | 公钥可以公开，用来推导地址或验证签名 | 地址 = 从公钥进一步处理的短标识，不是身份本身 |
| **Private Key** | 账户控制权本身，泄漏即失去所有权 | 不能截图、上传云盘、粘贴给网页、发给 AI 工具 |
| **Signature** | 私钥对消息生成的授权证明 | 不是"登录弹窗"，是对具体消息/交易的授权 |
| **Merkle Tree** | 用哈希组织大量数据，少量 proof 就能验证数据属于整体 | 空投名单、轻客户端、状态证明的基础 |

**第一性原理：**
> 链上身份不是由平台发给你的，而是由你能否证明自己控制某个私钥决定的。

**三条铁律：**
```
私钥是控制权  — 丢失 = 失去控制，泄漏 = 别人获得控制
签名是授权证据 — 签名内容必须能让人读懂，不能只展示一串数据
哈希是承诺    — 不能还原原文，但能验证数据有没有被改过
```

**Hash 在链上的常见用途：**
- 交易哈希 → 定位一笔交易
- 区块哈希 → 定位一个区块
- 合约字节码哈希 → 检查部署代码是否与源码一致
- 数据承诺 → 先公布哈希，之后公开原文让人验证没被改过

**私钥管理基本规则：**
```
❌ 不截图
❌ 不上传云盘
❌ 不粘贴给网页
❌ 不发给 AI 工具或客服
❌ 不放进代码仓库和 .env.example
❌ 不用主钱包测试陌生 dApp
```

**Signature 最容易出问题的地方：**
用户看不懂自己签了什么。产品应展示：
- 签名目的
- 域名 / 链 ID
- 合约地址
- 额度
- 过期时间
- 风险提示

**Merkle Tree 的工作方式：**
```
数据块 A → Hash(A)
数据块 B → Hash(B)     → Hash(Hash(A)+Hash(B)) → Merkle Root
数据块 C → Hash(C)
数据块 D → Hash(D)     → Hash(Hash(C)+Hash(D)) → Merkle Root
                           ↓
                        最终 Root（上链记录）
```

用户只需 Merkle proof（少量哈希路径）即可证明"我的数据在这批里"，无需下载全部数据。

**AI × Web3 视角：**
- 模型可以解释"这笔授权在批准某个 token 额度"
- 但不能替用户保管私钥
- 不能在用户不理解内容时推动签名
- AI 输出本身可能需要被签名/哈希/记录，作为审计证据

**最小实践：签名观察练习**
1. 创建一个测试钱包（不放真实资产）
2. 在测试环境发起一次普通转账或消息签名
3. 在区块浏览器/钱包界面观察：地址、交易哈希、签名提示、链 ID
4. 写下哪些信息可公开，哪些绝不能公开
5. 比较"签名消息"与"发送交易"的区别
